Antes de começarPré-requisitos

  • Docker com o plugin Compose (o comando docker compose, não docker-compose)
  • Um cliente MQTT para simular o ESP32 — mosquitto-clients (recomendado) via sudo apt install mosquitto-clients, ou paho-mqtt via pip install paho-mqtt

Confirme que as duas ferramentas respondem:

bash
docker compose version
mosquitto_pub --version

ContextoPipeline de uma leitura

O caminho percorrido por um pacote de telemetria, do dispositivo até o banco. Repare que a mesma ACL é consultada duas vezes: uma para autorizar a publicação da estação, outra para autorizar a entrega ao subscriber.

 ESP32
   │
   │  PUBLISH sensores/telemetria/esp32_teste_01
   │  {"capturado_em": "...", "temperature": 29.3, ...}
   ▼
 mosquitto                         FastAPI
   │──── POST /api/mqtt/auth ────► verifica username + bcrypt hash
   │◄─── 200 OK ─────────────────  (dispositivo autenticado)
   │
   │──── POST /api/mqtt/acl ─────► verifica regras_acl
   │     acc=2, topic=sensores/    (publish autorizado para %u)
   │◄─── 200 OK ─────────────────
   │
   │  mensagem aceita, enfileirada para entrega
   │
   │──── POST /api/mqtt/acl ─────► verifica regras_acl
   │     acc=1, topic=sensores/    (delivery autorizado para #)
   │◄─── 200 OK ─────────────────
   │
   │  entrega ao subscriber
   ▼
 subscriber
   │  1. deserializa JSON
   │  2. valida campos obrigatórios
   │  3. busca dispositivo_id por username_mqtt
   │
   │  INSERT INTO leituras
   │    (dispositivo_id, capturado_em, recebido_em, ...)
   ▼
 PostgreSQL — tabela leituras
   ┌─────────────────────┬──────────────────────────┐
   │ capturado_em        │ recebido_em              │
   │ (timestamp device)  │ (timestamp servidor)     │
   ├─────────────────────┼──────────────────────────┤
   │ 2026-07-13 17:00 Z  │ 2026-07-13 17:14 Z       │
   └─────────────────────┴──────────────────────────┘
          └── diferença indica latência ou período offline

Rejeições na pipeline

PontoCondiçãoResultado
authcredenciais inválidas ou dispositivo inativobroker recusa a conexão (CONNACK)
acl (publish)tópico diferente do própriobroker descarta silenciosamente
acl (delivery)regra de leitura ausente no bancobroker não entrega ao subscriber
subscribercampo faltando no JSONleitura descartada, erro logado
subscribercapturado_em inválidoinsert falha, rollback, erro logado
O publish rejeitado não avisa o cliente

Em MQTT 3.1.1 com QoS 0, o cliente não recebe notificação quando o broker bloqueia uma publicação. Para confirmar o bloqueio, verifique a ausência da leitura no banco.

Etapa 1Configurando as variáveis de ambiente

A stack lê as credenciais de infra/.env. Copie o exemplo e preencha os valores:

bash
cd infra/
cp .env.example .env

Para desenvolvimento local, os valores padrão do .env.example já funcionam com o seed de teste — basta descomentá-los.

Nunca commite o .env

O arquivo já está no .gitignore. As credenciais de teste deste guia são públicas por design e servem apenas para desenvolvimento local.

A senha do subscriber é gravada no banco

O valor de MQTT_SUBSCRIBER_PASSWORD definido aqui é o mesmo que o seed grava no banco durante a inicialização. Se você alterá-lo depois que a stack já subiu, recrie o volume com docker compose down -v para o seed rodar de novo com a senha nova — caso contrário o subscriber conecta com uma senha que não bate com o hash e a autenticação falha silenciosamente.

Etapa 2Subindo a stack

bash
cd infra/
docker compose up -d --build

A flag --build garante que as imagens da API e do subscriber sejam reconstruídas. Nas execuções seguintes, sem alteração de código, use apenas docker compose up -d.

Aguarde alguns segundos e confira se os quatro contêineres estão de pé:

bash
docker compose ps
saída esperada
NAME                     STATUS
observatorio_db          running
observatorio_api         running
observatorio_broker      running

API de autenticação

Credenciais válidas devem retornar 200:

bash
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://localhost:8000/api/mqtt/auth \
  -H "Content-Type: application/json" \
  -d '{"username": "esp32_teste_01", "password": "token_123"}'

E credenciais inválidas devem retornar 401:

bash
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://localhost:8000/api/mqtt/auth \
  -H "Content-Type: application/json" \
  -d '{"username": "esp32_teste_01", "password": "senha_errada"}'

Etapa 4Simulando um dispositivo ESP32

O dispositivo de teste já vem cadastrado no seed:

CampoValor
username_mqttesp32_teste_01
tokentoken_123
tópicosensores/telemetria/esp32_teste_01

Publicando com mosquitto_pub

bash
mosquitto_pub \
  -h localhost -p 1883 \
  -u esp32_teste_01 -P token_123 \
  -t "sensores/telemetria/esp32_teste_01" \
  -m '{"capturado_em":"2026-06-26T10:30:00Z","temperature":28.5,"humidity":72.1,"pm1_cf":12,"pm25_cf":18,"pm10_cf":24,"pm1_atm":11,"pm25_atm":17,"pm10_atm":23}'

Alternativa em Python

python
import paho.mqtt.client as mqtt
import json

payload = {
    "capturado_em": "2026-06-26T10:30:00Z",
    "temperature": 28.5,
    "humidity": 72.1,
    "pm1_cf": 12, "pm25_cf": 18, "pm10_cf": 24,
    "pm1_atm": 11, "pm25_atm": 17, "pm10_atm": 23,
}

client = mqtt.Client(mqtt.CallbackAPIVersion.VERSION2)
client.username_pw_set("esp32_teste_01", "token_123")
client.connect("localhost", 1883)
client.publish("sensores/telemetria/esp32_teste_01", json.dumps(payload))
client.disconnect()

Etapa 5Verificando a ingestão

Depois de publicar, o subscriber deve registrar a leitura:

bash
docker compose logs subscriber
saída esperada
... INFO Leitura persistida: esp32_teste_01 @ 2026-06-26T10:30:00Z

E a linha deve estar no banco:

bash
docker exec -it observatorio_db psql -U admin_ufpa -d observatorio \
  -c "SELECT capturado_em, recebido_em, temperature, humidity, pm25_cf FROM leituras ORDER BY recebido_em DESC LIMIT 5;"
saída esperada
       capturado_em        |         recebido_em          | temperature | humidity | pm25_cf
---------------------------+------------------------------+-------------+----------+---------
 2026-06-26 10:30:00+00    | 2026-06-26 13:45:01.234+00   |        28.5 |     72.1 |      18
Por que dois timestamps

capturado_em vem do dispositivo; recebido_em é gravado pelo servidor. A diferença entre os dois revela a latência de entrega — ou quanto tempo a estação passou offline antes de reenviar.

Etapa 6Testando a rejeição de acesso

O broker não aceita conexões anônimas nem publicações em tópicos alheios. Vale confirmar as duas barreiras.

Conexão sem credenciais

bash
mosquitto_pub -h localhost -p 1883 \
  -t "sensores/telemetria/esp32_teste_01" \
  -m "teste" 2>&1
saída esperada
Connection error: Connection Refused: not authorised.

Publicação em tópico alheio

Aqui a estação está autenticada, mas tenta publicar no tópico de outro dispositivo:

bash
mosquitto_pub \
  -h localhost -p 1883 \
  -u esp32_teste_01 -P token_123 \
  -t "sensores/telemetria/outro_dispositivo" \
  -m "teste"
A rejeição é silenciosa no cliente

Em MQTT 3.1.1 com QoS 0, o broker descarta a mensagem sem avisar o cliente: o comando termina com exit 0 e sem saída. A rejeição aparece no log da API (401) e é confirmável pelo banco — nenhuma linha nova deve surgir em leituras.

bash
docker exec -it observatorio_db psql -U admin_ufpa -d observatorio \
  -c "SELECT COUNT(*) FROM leituras;"

O contador deve permanecer igual ao de antes da tentativa.

Etapa 7Acompanhando os logs em tempo real

Para depurar o fluxo completo, deixe dois terminais abertos.

Terminal 1 — subscriber

bash
cd infra && docker compose logs -f subscriber

Terminal 2 — broker

bash
cd infra && docker compose logs -f mosquitto

Etapa 8Parando a stack

bash
cd infra && docker compose down

Os dados ficam preservados no volume db_data. Para resetar tudo do zero:

bash
docker compose down -v
down -v apaga o banco

Use somente quando quiser recomeçar do estado limpo — por exemplo, depois de alterar init.sql ou seed.sql, que só rodam na criação do volume.
